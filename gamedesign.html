Spent

Game Design Case Study — Momentum-Based FPS

A fast-paced FPS built around fluid movement, player expression, and maintaining momentum through chained mechanics. This document covers the design decisions behind the systems and how thinking like a designer shaped the technical implementation.

Overview

Spent is a momentum-driven FPS designed around fluid traversal and high-speed combat. The player navigates an arena using mobility-based mechanics while eliminating waves of enemies and maintaining flow state. Every system was designed with the question: "does this preserve or break momentum?"

Design Pillars

Flow Motion

Maintain constant momentum and responsiveness. No ability should feel like a full stop.

Skill Expression

Reward precision, timing, and chaining mechanics. The skill ceiling should feel meaningful.

Stylized Immersion

Distorted visuals and feedback reinforce the robotic identity of the character.

Challenge Clarity

Fast-paced gameplay that stays readable and fair — no cheap deaths.

Core Gameplay Loop

The player navigates the arena using mobility abilities, eliminates enemy waves, and chains mechanics to maintain speed and control. Each wave is designed to create pressure that rewards movement over static play.

System Design

  • Movement: Wall running, sliding, grappling, and dashing — each built as an independent component sharing a velocity state.
  • Combat: Precision shooting combined with movement positioning — standing still is a disadvantage.
  • Abilities: Ground slam, environmental interaction — abilities that reward aerial play.
  • Enemies: Ranged, melee, and flying types with FOV-based detection — variety that demands different movement responses.

Key Design Decisions

The core decision that shaped everything: instead of overriding movement during abilities, every system modifies existing velocity. Wall running adds forward force. Sliding adds a downward impulse. The grapple accelerates rather than teleports. This means mechanics chain naturally — a slide into a grapple into a wall run all feel connected because they share the same velocity state.

This was a deliberate design choice that required a central state machine (PlayerMovementAdvancedFinished) to coordinate which system owns velocity at any given moment. The result is a movement system that feels expressive without adding complexity for the player.

The arena is designed as a movement puzzle — vertical space, walls at useful angles, grapple points placed to reward creative traversal rather than just providing shortcuts.

Design → Code Connection

Each design pillar has a direct code equivalent:

  • Flow Motion → SmoothlyLerpMoveSpeed() — speed transitions are lerped, not snapped, so state changes feel fluid.
  • Skill Expression → Wall memory system — you can't re-use the same wall without leaving it first, so chaining requires intentional movement.
  • Challenge Clarity → FOV + line-of-sight checks on enemies — they never shoot through walls or from off-screen.
  • Scalability → All enemy behaviour is component-based — a flying shooter is just FlyingEnemy + EnemyShooting + EnemyHits with no duplicated logic.

Testing & Iteration

  • Movement feel rated as the strongest element during playtesting — the momentum system was validated early and kept stable
  • Enemy detection ranges and fire rates were tuned iteratively to ensure fairness without reducing difficulty
  • UI clarity improved after feedback — status indicators and ammo counter repositioned for better readability at speed
  • Tutorial dialogue system added after testing showed new players weren't discovering mechanics organically
